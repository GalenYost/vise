## Building
- [nob.h](https://github.com/tsoding/nob.h) by Tsoding is being used for this project as a build system
- build recipe is found in nob.c file in the project root
- Makefile is present to build with one simple command

### nob.c
There are a few details to clarify.
1. constants (define's):
- BUILD_FOLDER is created and used to store build / runtime logs, as well as final executable
- EXEC_NAME is used to create an executable
- COMPILER_NAME is basically a part of output, kind of an "official" name you could say
2. struct's:
- Command - single command, has a `name`, `signature` and `description`, used as an instance to describe compiler commands for visual output for help message
- Commands - dynamic array of `Command` structs, nothing else to say
3. functions, they are self-explanatory, but still:
- `ends_with` - returns a result of `strcmp` on string, to match its suffix
- `collect_sources` - appends all of the matching files (ends_with .c) to Nob_Cmd instance
- `command` - creates a `Command` instance at runtime
- `print_help` - takes a `Commands` array and displays all of its children
- `build_compiler` - executes a `cc` command with collected sources into a `BUILD_FOLDER`/`EXEC_NAME` path
- `run_compiler` - executes the compiler binary
- `main` - entry point, defines available compiler commands

## Code documentation
Not going to explain every single detail, just some short general moments for understanding.

### AST
ast.h header contains only typedef's for AST-related things.
This includes:
1. Token struct, structure that lexer uses to return parsed tokens
2. Node struct, structure that parser uses to return a result of parsing
3. DataType struct, structure that is being used in Node structure as a way to give more details on what kind of variable we are holding

### Lexer
Header file only exposes `new` and `next` function to create Lexer instance and iterate through Lexer's Token's accordingly.
Structure itself contains a `start` point for a Token, `current` point where Lexer is currently at, `line` and `col` to give the exact location and `arena` for memory storing, things like `StringStream` for example.
Implementation contains `static` helper functions that simplify code readability, they should be straight-forward unless an additional comment is left nearby to explain function better.

### Parser
The Parser implements a Recursive Descent parsing algorithm. It consumes `Token`s generated by the Lexer and constructs an Abstract Syntax Tree (AST). The AST is composed of `Node` structures (defined in `ast.h`). The parser is responsible for syntactic validation, ensuring that the source code adheres to Vise's grammar rules before passing the AST down to the type checker and compiler phases.

### Memory management
The compiler utilizes an Arena Allocator (`arena.c` / `arena.h`) to handle memory efficiently. Since a compiler typically allocates many small objects (like AST nodes and Tokens) that all share the same lifetime (the duration of the compilation process), an Arena allows us to:
- Allocate memory extremely fast (just incrementing a pointer).
- Avoid manual `free()` calls for individual nodes.
- Free all allocated memory at once when compilation finishes or the arena is destroyed.

### Utilities
The `utils.c` and `utils.h` files contain common helper functions utilized across the compiler's codebase. This includes:
- File I/O operations (e.g., reading source files into memory).
- String manipulation and formatting.
- Dynamic array implementations (like the one used for the `Commands` array in the build system).
- Generic error reporting and logging macros.
